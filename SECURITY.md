# Security Policy

## Поддерживаемые версии

Проект распространяется по принципу «rolling release» — поддерживается только последний коммит в ветке `main`. Старые версии не получают патчей.

| Версия | Поддержка |
|---|---|
| `main` (последний коммит) | ✅ |
| Любая прошлая | ❌ |

## Что считается уязвимостью

В контексте этого проекта уязвимостью считается **любая ситуация, в которой пользователь рискует утечкой персональных данных или компрометацией учётных записей**:

- 🔴 **Critical.** Утечка содержимого `profile.py`, `hh_session.json`, `.env` в публичный репозиторий через коммит, лог, кеш сборки или артефакт.
- 🔴 **Critical.** Жёсткое прописывание (хардкод) реальных ФИО, email, телефонов, токенов в любом файле, попадающем в git.
- 🔴 **Critical.** Передача `ANTHROPIC_API_KEY` или сессии HH в третьесторонние сервисы без явного согласия пользователя.
- 🟠 **High.** XSS / HTML-injection в дашборде (например, через имя работодателя или текст сообщения от рекрутёра).
- 🟠 **High.** SQL-injection в `metrics.py` (хотя сейчас используются параметризованные запросы — нужно следить, чтобы это не сломали будущим коммитом).
- 🟡 **Medium.** Утечка пути к локальным файлам или PID в логах, попадающих в публичный канал (issue, gist).
- 🟡 **Medium.** Race condition при параллельном запуске нескольких воркеров, ведущая к двойной отправке отклика/ответа.
- 🟢 **Low.** Прочие предсказуемые слабости — например, отсутствие rate-limit на `/api/*`-эндпоинтах дашборда, даже несмотря на то, что он привязан к `127.0.0.1`.

Если не уверены, считается ли что-то уязвимостью — пишите всё равно.

## Как сообщить об уязвимости

**Не открывайте публичный issue.** Используйте один из приватных каналов:

### Способ 1 (рекомендуется): GitHub Security Advisories

1. Откройте <https://github.com/semernyakov/hh-auto-apply/security/advisories/new>
2. Заполните Title и Description (см. шаблон ниже).
3. Прикрепите PoC (proof of concept), если есть.
4. GitHub автоматически создаст приватное обсуждение между вами и maintainer.

### Способ 2: GitHub Private Vulnerability Reporting

Если функция Security Advisories недоступна в вашей роли — нажмите **«Report a vulnerability»** на вкладке Security самого репозитория.

### Шаблон отчёта

```
## Summary
[Одно предложение: что и где]

## Severity
[critical / high / medium / low]

## Affected files / endpoints
[например: dashboard.py:849, /api/chat-history]

## Steps to reproduce
1. ...
2. ...
3. ...

## Proof of concept
[curl-команда, скриншот, fragment кода]

## Impact
[Что злоумышленник может сделать, какие данные пострадают]

## Suggested fix
[Опционально — если есть идея]

## Reporter
[Имя/ник и контакт для связи, опционально]
```

## Чего НЕ нужно делать

- 🚫 **Не публикуйте PoC** в открытых каналах (Twitter, Telegram, public issues), пока вопрос не решён.
- 🚫 **Не тестируйте уязвимости на чужих аккаунтах HH.ru** без согласия владельца.
- 🚫 **Не используйте найденную уязвимость** для обогащения, шантажа, нанесения ущерба пользователям или платформе HH.ru.

Нарушение этих правил снимает с maintainer обязательства по обработке отчёта и может привести к жалобе через GitHub Trust & Safety.

## Сроки реагирования

| Шаг | SLA |
|---|---|
| Подтверждение получения отчёта | ≤ 5 рабочих дней |
| Первичный анализ и категоризация | ≤ 10 рабочих дней |
| Фикс для **Critical** | ≤ 14 дней с момента подтверждения |
| Фикс для **High** | ≤ 30 дней |
| Фикс для **Medium / Low** | best effort, без жёстких сроков |
| Публичное раскрытие | После релиза фикса или по согласованию с репортёром, не ранее 30 дней с подтверждения (coordinated disclosure) |

Если maintainer не отвечает в течение 14 дней — можете подсветить через `@`-mention в Security Advisory; если и тогда нет ответа — допустимо публичное раскрытие через 90 дней с момента отчёта.

## Признание

С согласия репортёра, имя/ник может быть упомянуто в:

- changelog соответствующего релиза,
- разделе «Acknowledgments» в README,
- описании GitHub Security Advisory.

Денежного bug-bounty проект не предлагает (open source без коммерции).

## Out of scope

Следующее **не считается** уязвимостью проекта (не отчётом для этой политики, а нормальным issue или вообще ожидаемым поведением):

- Блокировка аккаунта HH.ru — см. [DISCLAIMER.md](DISCLAIMER.md), это известный риск.
- Изменение DOM-структуры HH.ru, ломающее селекторы Playwright — это поведение HH, не уязвимость.
- Цена Anthropic API, превышение бюджета — пользователь контролирует ключ и интервалы.
- Capability «вытащить свой `profile.py` через дашборд» — это и есть его данные, доступ к ним подразумевается.
- Отсутствие 2FA на дашборде — он биндится на `127.0.0.1`, локальный режим. Если выставляете наружу — это уже ваша конфигурация.
