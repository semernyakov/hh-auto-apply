# Contributing

Спасибо за интерес к `hh-auto-apply` 🙌 Любой вклад — от опечатки в README до новой фичи — приветствуется.

## Чем можно помочь

- 🐛 **Багрепорты.** Если что-то сломалось — откройте issue с шагами воспроизведения и логом из `/tmp/hh_auto_*.log`.
- 💡 **Идеи.** Предложите фичу через issue с тегом `enhancement`. Объясните задачу, которую она решает.
- 🧹 **Качество кода.** Рефакторинг, типизация (`mypy`), линт-фиксы.
- 📚 **Документация.** Опечатки, неточности, недостающие примеры в README/SETUP_GUIDE.
- 🌍 **Локализация.** English-версия README уже частично пригодилась бы.
- 🧪 **Тесты.** Сейчас их нет — любой `pytest`-каркас будет в плюс.

## Workflow

1. **Сначала issue, потом PR.** Если фича крупная — обсудите дизайн в issue до того, как писать код. Для опечаток / небольших правок issue не нужен.
2. **Форкните репозиторий**, создайте ветку из `main`:
   ```bash
   git checkout -b feat/short-description
   ```
3. **Сделайте изменения** маленькими, осмысленными коммитами (см. ниже).
4. **Откройте Pull Request** против `main`. Опишите:
   - Что меняется и зачем.
   - Как проверять (шаги).
   - Связанные issues (`Closes #N`).

## Установка для разработки

```bash
git clone https://github.com/<your-fork>/hh-auto-apply.git
cd hh-auto-apply
python3 -m venv .venv && source .venv/bin/activate
pip install --upgrade pip
pip install playwright python-dotenv anthropic fastapi uvicorn
pip install ruff mypy   # dev-инструменты
python -m playwright install chromium
cp profile.example.py profile.py   # тестовые данные, в git НЕ пушить
```

## Стиль кода

- **Python 3.10+**, типизация по PEP 484 где возможно.
- **Линт:** `ruff check .` — не должно быть ошибок.
- **Форматирование:** код подчиняется правилам Ruff (`E`, `F`, `I`, `W`). Двойные кавычки.
- **Имена:** snake_case для функций/переменных, UPPER_SNAKE_CASE для констант.
- **Комментарии** — по делу, на русском там, где остальной комментарий русский. Не дублируйте код.
- **Никакого магического числа:** константы выносите наверх модуля или в `profile.py` (если зависит от пользователя).

## Commit-сообщения

Один коммит = одна логическая единица. Сообщение по схеме [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>: <summary>

<body — что и зачем, не «как»>
```

Типы: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `style`.

Примеры:

```
feat: add Telegram notification on positive_signal events

При появлении приглашения на интервью или запроса контактов отправляем
Telegram-сообщение через bot API. Закрывает #42.
```

```
fix(reply): не отвечать в чате если последнее сообщение от соискателя

Раньше эвристика «нет author field → исходящее (мы)» падала на собственные
голосовые сообщения. Теперь смотрим side ('right'/'left') из HH DOM.
```

## Чек-лист перед PR

- [ ] Локально работает (`make start` → дашборд открывается без ошибок).
- [ ] `ruff check .` — чисто.
- [ ] **`profile.py` НЕ в коммите.** Проверка: `git status` не должен показывать `profile.py`.
- [ ] **Никаких личных данных** в коде, тестах, фикстурах. Запустите:
  ```bash
  git diff --staged | grep -iE "ваше_фио|ваш_email|ваш_телефон|sk-ant-[a-zA-Z0-9]"
  ```
  Должно быть пусто.
- [ ] Описание PR объясняет «зачем», а не только «что».
- [ ] Если затрагивается dashboard UI — проверено в браузере (Chromium/Firefox).
- [ ] Документация (README/SETUP_GUIDE) обновлена, если поведение изменилось.

## Что я НЕ приму в PR

- Изменения, нарушающие приватность пользователей: хардкод реальных ФИО, телефонов, email, токенов.
- Функции для **массового spam'а** или обхода защитных механизмов HH (капча, anti-bot fingerprint).
- Сбор и передача данных третьих лиц (рекрутёров, работодателей) за пределы локальной БД пользователя.
- Зависимости с несовместимой лицензией (GPL для проекта под MIT).
- Огромные «refactor everything» PR без обсуждения.

## Уязвимости

Если нашли security-issue (особенно утечку приватных данных в коде или истории) — **не открывайте публичный issue**. Используйте GitHub Security Advisories — см. [SECURITY.md](SECURITY.md).
